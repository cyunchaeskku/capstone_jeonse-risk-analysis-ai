# Architecture

## 목적

이 문서는 시스템 전체 구조와 모듈 간 책임 경계를 정의하는 기준 문서다.
구현 전에 모든 구성원이 같은 시스템 모델을 공유하는 것을 목표로 한다.

## 시스템 목표

- 전세계약 관련 입력 정보를 받아 위험 요소를 분석한다.
- 분석 결과를 사용자에게 설명 가능한 형태로 제공한다.
- 법률 및 판례 기반 추가 질의응답을 지원한다.

## 상위 흐름

1. 사용자가 매물 정보와 문서를 입력한다.
2. 문서 처리 파이프라인이 입력을 정규화한다.
3. 위험 분석 엔진이 규칙 기반 평가를 수행한다.
4. 설명 생성 계층이 분석 결과를 자연어로 정리한다.
5. 질의응답 계층이 법률 및 판례 근거 기반 답변을 생성한다.

## 모듈 경계

### Frontend

- 입력 폼, 업로드 UI, 분석 결과 화면, 질의응답 UI를 담당한다.
- 백엔드 API 응답을 그대로 보여주지 않고 사용자 친화적 형태로 재구성한다.

### Backend API

- 프론트엔드 요청을 수신하고 각 도메인 서비스로 라우팅한다.
- 인증, 검증, 오류 응답, 분석 작업 오케스트레이션을 담당한다.

### Document Pipeline

- 계약서, 등기부등본, 공공 데이터 등 여러 입력 소스를 텍스트와 구조화 데이터로 변환한다.
- 원문 보관 정책, 추출 결과, 정규화 규칙의 기준점을 관리한다.
- 법령 원문 소스로 `legalize-kr` 저장소를 활용할 수 있다.

### Risk Analysis Engine

- 규칙 기반으로 위험 요소를 평가한다.
- 개별 규칙 결과와 종합 위험도를 생성한다.
- 설명 생성 계층과 분리하여 deterministic 판단 기준을 유지한다.

### Legal QA

- 질문을 먼저 분류한 뒤, 단순 질문은 검색 없이 답변하고 법령 질문만 문서 검색으로 보낸다.
- 법률 및 판례 문서를 검색하고 질문 관련 근거를 회수한다.
- 회수한 문서를 기반으로 grounded answer를 생성한다.
- 위험 판정 엔진의 판단 로직을 대체하지 않는다.
- `korean-law-mcp`는 법령 검색 및 조문 조회 인터페이스 후보로 본다.
- 초기 벡터 검색 경로는 `vectorDB/laws_faiss`의 로컬 FAISS 인덱스를 사용한다.

### Storage

- PostgreSQL: 사용자 입력, 분석 이력, 정형 결과 저장
- FAISS: 법률/판례/가이드 문서의 벡터 인덱스 저장

## 주요 경계 원칙

- 위험 판단은 규칙 엔진이 담당한다.
- LLM은 설명과 질의응답에 사용하며, 최종 위험 판정의 단독 근거가 되지 않는다.
- 법률 QA는 참고 정보 제공 계층이며 위험도 산정 규칙과 분리한다.
- 외부 법령 저장소와 MCP 서버는 데이터 공급 계층으로 취급하며, 내부 도메인 모델과 직접 결합하지 않는다.

## 주요 오픈 이슈

- 초기 버전에서 비동기 분석 작업이 필요한지 여부
- 문서 OCR 정확도 기준과 fallback 전략
- 종합 위험도 산정 방식의 점수화 여부
